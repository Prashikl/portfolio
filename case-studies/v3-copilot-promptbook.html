AI Tool Design · Reference Architecture

A promptbook that turns Copilot into a domain expert.

Most prompt collections are vibes — clever turns of phrase that fall apart the moment a teammate inherits them. I built a promptbook with the discipline of a design system: 15 playbooks, 4000+ prompts, 20+ evidence-backed sources, and a reference architecture that refuses to ship a claim without a citation.

i
The premise

Prompts are not the product

A clever prompt is a parlour trick. A system of prompts — one that hands a teammate the same expert behaviour, day after day, with the same evidence trail — is a piece of design infrastructure.

The promptbook started as my own working notes for AI-accelerated design and research. It grew into a shared reference because the same complaint kept landing in my inbox: "I tried Copilot for synthesis and it made things up." The model wasn't broken. The scaffolding around it was missing.

ii
Architecture

Four layers, each with a single job

L1

Sources of truth

20+ evidence-backed sources — research papers, design system docs, internal playbooks, named expert interviews. Every prompt cites the source layer it's allowed to draw from. If the source doesn't say it, the prompt cannot claim it.

L2

Operating principles

Eight non-negotiables that govern every prompt — citation discipline, refusal protocols, domain boundaries, tone, and the rule that ambiguity must surface before output.

L3

Playbooks

15 domain-bounded playbooks — research synthesis, content audits, accessibility review, competitive teardowns, journey mapping, design critique, spec drafting, and more. Each playbook is a stack of prompts that share a goal.

L4

Prompts

4000+ composable prompts. Versioned. Tagged by playbook, role, and source layer. A teammate copies a prompt and gets the same shape of output I would — including the same refusals when the source layer doesn't support a claim.

Architecture rule

A prompt without a source layer is a prompt without a conscience. Don't ship it.

iii
Operating principles

The eight non-negotiables

01

Cite or refuse

Every claim must be tied to a source layer. If no source supports it, the prompt must refuse rather than fabricate.

02

Surface ambiguity early

When the input is underspecified, the prompt asks before it answers. Hallucinated assumptions are a failure mode, not a feature.

03

Domain boundaries are hard

A research-synthesis prompt does not write specs. A spec prompt does not invent research. Each playbook stays in its lane — the user composes lanes, not the model.

04

Output shape is part of the contract

Every prompt declares its output schema. A teammate inheriting it gets the same shape — not a "creative interpretation" each session.

05

Tone is set by the playbook

Critique prompts are direct. Synthesis prompts are conservative. Drafting prompts are generative. Tone is an architectural choice — not a per-prompt whim.

06

Versioning, not vibes

Prompts are versioned like code. Improvements ship as v2 with a changelog — not as a silent rewrite that breaks downstream playbooks.

07

Human-in-the-loop, by design

Every playbook has at least one mandatory human checkpoint. AI accelerates; humans accept. Skipping the checkpoint is a usage error.

08

Failure modes are documented

Each playbook lists what it cannot do, and what it commonly gets wrong. Knowing the edges is the difference between a tool and a trap.

Lesson

A prompt collection is a notebook. A promptbook is a system. The difference is whether your teammate can inherit your judgement, or only your wording.

vi
Why it matters

AI tools are a design surface

The interesting question is not "what can the model do." The interesting question is "what scaffolding makes the model behave like a colleague we'd actually trust." That scaffolding is design work — content design, system design, governance design.

The promptbook is a small example of a much larger discipline coming into view: designing the conditions under which a model is allowed to speak on our behalf.
